# Terminal UI

Документ фиксирует договоренности по терминальному интерфейсу `ai-usage`, прежде всего по loader-индикации во время получения usage/limits.

---

## Цель

Пользователь не должен видеть "тишину" после запуска команды, если получение данных занимает несколько секунд.

Terminal UI должен:

- показывать, что работа идет
- показывать отдельные ожидаемые источники данных
- не обещать точный прогресс, если источник не отдает реальный percent complete
- не смешивать UI-логику с provider-логикой
- корректно завершаться при успехе и ошибке

---

## Loader

Основной loader — estimated progress bar.

Он отражает не фактический процент внутренних шагов провайдера, а визуальную модель ожидания:

```text
▰▰▰▰▰▰▰▱▱▱ 65% codex-cli
```

Ключевое правило:

- `100%` показывается только после реального ответа источника

До ответа loader может дойти примерно до `90%`, затем сильно замедлиться и ползти до верхнего лимита ожидания, например `95%`.

Когда ответ получен, loader быстро, но не мгновенно, доходит до `100%`.

---

## Общая Рамка Ответа

Каждый терминальный ответ `ai-usage`, включая help и ошибки, должен быть оформлен общей рамкой.

Первая строка:

```text
=-=-=-=-=-=-=-=-=-=-=-=-= AI USAGE =-=-=-=-=-=-=-=-=-=-=-=-=
```

Финальная строка зависит от результата:

```text
=-=-=-=-=-=-=-=-=-=-=-=-=-= DONE =-=-=-=-=-=-=-=-=-=-=-=-=-=
=-=-=-=-=-=-=-=-=-=-=-=-=-= PART =-=-=-=-=-=-=-=-=-=-=-=-=-=
=-=-=-=-=-=-=-=-=-=-=-=-=-= FAIL =-=-=-=-=-=-=-=-=-=-=-=-=-=
```

Статусы:

| Статус | Значение |
| --- | --- |
| `DONE` | все запрошенные источники успешно отдали данные или корректное unavailable-состояние |
| `PART` | часть источников отдала данные, часть завершилась ошибкой |
| `FAIL` | команда не смогла получить пригодный результат |

Главное визуальное правило:

- верхняя и нижняя строки должны начинаться и заканчиваться на `=`
- допускается разница в 1 символ по длине, если это дает более аккуратное завершение строки
- верхняя и нижняя строки визуально толще внутренних разделителей

## Заголовки Разделов

Каждый источник данных выводится отдельным блоком.

Заголовок блока:

```text
            ~~~~~~~~~~ CURSOR-API2 ~~~~~~~~~~
            ~~~~~~~~~~ CODEX-CLI ~~~~~~~~~~~~
            ~~~~~~~~~~ CLAUDE-CLI ~~~~~~~~~~~
```

Правила:

- внутренние заголовки короче верхней и нижней рамки
- внутренние заголовки используют `~`
- левый отступ внутренних заголовков фиксированный
- правый край может немного отличаться из-за разной длины названия источника
- важнее выровнять левый край, чем добиться идеальной симметрии

Под заголовком блока выводится либо текущий loader, либо готовый результат источника.

Пример целевого вида:

```text
=-=-=-=-=-=-=-=-=-=-=-=-= AI USAGE =-=-=-=-=-=-=-=-=-=-=-=-=

            ~~~~~~~~~~ CURSOR-API2 ~~~~~~~~~~

Cursor usage:
Plan usage: 120 / 2000
Billing cycle resets: 2026-07-01

            ~~~~~~~~~~ CODEX-CLI ~~~~~~~~~~~~

▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▱▱▱▱▱▱▱▱ 84%

            ~~~~~~~~~~ CLAUDE-CLI ~~~~~~~~~~~

▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▰▱▱▱▱▱ 90%

=-=-=-=-=-=-=-=-=-=-=-=-=-= DONE =-=-=-=-=-=-=-=-=-=-=-=-=-=
```

## Источники И Скорость

Скорость loader-а зависит от типа provider method.

Типы:

| Тип | Ожидаемая скорость | Примеры |
| --- | --- | --- |
| `local-files` | самая быстрая | будущие чтения локальных telemetry/transcript файлов |
| `api` | средняя | `cursor-api2` |
| `cli` | самая медленная | `codex-cli`, `claude-cli` |

Эти скорости являются эвристикой, а не измерением реального прогресса.

Provider method сообщает только реальные события:

- работа началась
- работа завершилась успешно
- работа завершилась ошибкой или unavailable-состоянием

## Параллельная Модель

Если пользователь ожидает данные от нескольких источников, loader должен показывать несколько строк.

Пример:

```text
▰▰▰▰▰▰▰▱▱▱ 65% codex-cli
▰▰▰▰▱▱▱▱▱▱ 40% claude-cli
▰▰▰▰▰▰▰▰▱▱ 80% cursor-api2
```

Если отображаются несколько loader-ов одновременно, сами запросы тоже должны выполняться параллельно. Последовательное выполнение с параллельной индикацией вводит пользователя в заблуждение.

Целевая модель:

```text
provider worker threads
        ↓
channel events
        ↓
cli event loop
        ↓
multi-loader renderer
```

Когда один источник готов, его loader завершается, а результат может быть показан пользователю, пока остальные источники продолжают работу.

Порядок вывода результатов должен быть отдельным UX-решением:

- можно печатать результат сразу по готовности
- можно сохранять фиксированный порядок провайдеров

Для ощущения скорости предпочтительнее печатать результат по готовности, но при этом ясно указывать имя источника.

## Unicode И Fallback

Основной визуальный вариант:

```text
▰▰▰▰▰▰▰▱▱▱ 65%
```

Это Unicode, не ASCII.

Риск считается допустимым для целевой аудитории, потому что пользователи CLI-инструментов для LLM с высокой вероятностью используют современные терминалы.

Fallback нужен для нестандартных сред.

Условия fallback:

- stdout не является TTY
- окружение не выглядит как UTF-8
- пользователь явно отключил расширенный вывод будущей настройкой

ASCII fallback:

```text
[#######---] 65%
```

## Цвет

Заполненная часть loader-а может быть цветной через ANSI color codes.

Цвет является дополнительным улучшением, а не обязательной частью смысла.

Правила:

- если цвет не поддерживается, строка все равно должна быть читаемой
- если stdout не TTY, ANSI color codes не должны попадать в вывод
- если установлен `NO_COLOR`, цвет должен быть отключен
- цвет не должен быть единственным способом понять состояние

## Ошибки И Завершение

При успехе:

- loader быстро доходит до `100%`
- строка очищается или заменяется на `done`
- затем печатается результат источника

При ошибке:

- loader останавливается
- строка становится `failed` или `unavailable`
- остальные источники продолжают работу
- общий exit code определяется правилами CLI, а не renderer-ом

При аварийном завершении:

- курсор должен быть возвращен
- временные loader-строки должны быть очищены или оставлены в читаемом состоянии

## Архитектурные Границы

Loader-логика относится к terminal UI, а не к provider-логике.

Целевое размещение:

```text
src/cli/mod.rs
  - парсит аргументы
  - запускает event loop terminal UI
  - вызывает общий сценарий получения данных
  - печатает финальный результат

src/infra/loader.rs
  - выбирает unicode/ascii
  - выбирает color/no-color
  - рисует один или несколько loader-ов
  - очищает строки
  - восстанавливает курсор

src/get_limits.rs
  - координирует provider methods
  - не форматирует terminal output

src/providers/*
  - получают данные
  - не знают про loader, ANSI, цвет, cursor movement
```

Provider method не должен напрямую рисовать loader.

Если нужна связь между provider execution и terminal UI, она должна идти через события или нормализованные статусы, а не через прямые вызовы UI из provider-файлов.

---

## MVP И Целевое Состояние

MVP может начинаться с одного общего loader-а или loader-а на текущий provider.

Целевое состояние:

- параллельный запуск выбранных provider methods
- отдельная loader-строка на каждый ожидаемый источник
- estimated progress по типу источника
- `100%` только по факту ответа
- Unicode `▰▱` по умолчанию при UTF-8 TTY
- ASCII fallback для небезопасных сред
